IrodsShare HSAccess usage¶

HSAccess implements a python interface to the iRODS access control system. This document describes usage of the low level python interface. There is also a high-level python interface HSAccessObjects.

Overview¶

To use this class, one establishes a connection via the HSAccess constructor. This class is a wrapper around native iRODS access control and does not create iRODS objects nor objects in HydroShare. Instead, it maintains a concept of access control for three kinds of objects:

  1. Users in iRODS
  2. Resources stored in iRODS
  3. Groups of users defined in HydroShare in django.

The first and second kinds of objects are physical entities. The third is contained solely inside the access control system.

Rationale and need¶

This module was created in order to extend the concept of protection available in iRODS. Currently in iRODS, creation of a user group is a privileged action. We wanted to create unprivileged notions of:

  • Creating a group.
  • Sharing a file with another user or group.

that can be used by any user and allows a very flexible and fluid form of data sharing similar to what is available in Dropbox, Google Drive, OneDrive, but acts at the filesystem level so that the resources under its control are usable as if they exist in a file system.

Architecture¶

IrodsShare is based upon a PostgreSQL database that is deployed in parallel to the iRODS database iCAT. It was found early in the design process that iCAT extensions are not sufficiently expressive to be used as a security and access control model of this type. Thus, a separate database was created. This database stores access control information for iRODS “resources”.

In order to accomplish this, the iRODS instance must be specially configured. All resources in iRODS must be group-writeable to a special iRODS group ‘HydroShare’ that contains all users of IrodsShare. IrodsShare imposes its protections on top of iRODS protections, but at all levels of iRODS, so that access through the command line, iRODS API, and iRODS REST services are all subject to this access control.

Theory of operation¶

This module acts on two levels:

  1. Controlling access to resources existing in iRODS.
  2. Controlling access through the django web interface.

It exists as a separate entity because these access control mechanisms must be consistent.

In iRODS, it is possible to define a method that is called before anything is done to a resource. This method allows one to deny access to unauthorized users. We utilize this method to invoke our own authorization system based upon our authorization database.

In Django, this library serves as a definitive source of access control information. Access is registered by Django, and read from this source as necessary. The module API includes listing functions that allow one to list resources by access.

Naming conventions¶

In this library, there are several strong naming conventions:

  • Methods starting with assert_ create and change records for an entity. These are idempotent and can be reused to change metadata after creating it.
  • Methods starting with share_ and unshare_ grant and remove privileges to an object.
  • Methods starting with get_ report on the state of the access control system.

Usage¶

After utilizing the HSAccess constructor, one gains access as a specific user. This user must be already created in iRODS and that user’s uuid is available by calling HSAccess.get_user().

When using HSAccess, every object is identified by a 32-byte Universal Unique Identifier (UUID). There are UUID namespaces for users, groups, and resources. Other objects, including folders, do not have UUIDs within the system. In general, when the system references a group, user, or resource, one must know the uuid. Listing functions allow one to easily recover the uuid for all relevant objects that one is allowed to access.

The use of UUIDs allows us to be quite flexible on naming of objects. For example, in IrodsShare, it is not an error for two groups or resources to have the exact same name. The UUIDs disambiguate between these.

For example, consider HSAccess.share_resource_with_user(). This requires two uuids: one identifying the user and another identifying the resource. A third uuid is that of the current user, which is contained inside the HSAccess object and need not be mentioned.

Many methods in HSAccess have optional parameters. For example, whenever appropriate, the user uuid is optional and defaults to that of the current user as defined when creating the object.

Sharing in IrodsShare¶

The sharing interface is the reason for the existence of IrodsShare. It has several unique properties, including:

  • Fluidity: sharing privileges are unprivileged and based upon the idea that one can delegate any privilege one has.
  • Cumulative privilege: sharing privileges are cumulative over all intents to share from all users. If two people share the same object with different privileges, the higher privilege wins.

Privileges¶

The privilege system in IrodsShare is based upon four distinct levels of privilege, including:

  • Owner: can do anything to an object, and can unshare it with a user as necessary.
  • Read/Write: can read or update an object, but cannot delete it or override other users’ sharing of it.
  • Read-Only: can read but not write the object.
  • Read-Only without sharing privilege: can read the object but cannot share the object with others.

The privilege system is based upon several policy rules:

  • Every object (resource and group) must have an owner. The initial owner is the creator.
  • The last owner cannot be removed from an object. But new owners can be assigned before removing the original creator as owner.
  • There can be multiple owners.
  • A user of the system can share an object only at the user’s own privilege level or below. E.g., a person with read/write privilege cannot share an object as owner, but can share with “read/write”, “read-only”, or “read-only without sharing” access. Obviously, a user with “read-only without sharing” cannot share the object.
  • A user can change the privilege on a share without notice, within these bounds.
  • A user that is a member of a group can share an object with the group. Group privilege applies to all members of the group.

For groups, there are extra meanings to the protections

  • Owner means that one can destroy the group and remove members.
  • Read/Write means that one can add members to the group.
  • Read-only means that one can only list members. There is no lower level of group access.

Instantaneous and invited privilege¶

There are two kinds of sharing in IrodsShare: instantaneous and invited.

  • Sharing of resources with either users or groups is instantaneous; the privilege appears immediately.
  • Sharing of groups is invited; the invited user must approve group membership in a separate confirmation.

The latter is a state-machine model of sharing, to wit:

  1. One user invites another to a group with HSAccess.invite_user_to_group().
  2. The user can:
    1. List invitations through HSAccess.get_group_invitations_for_user().
    2. Accept invitations through HSAccess.accept_invitation_to_group().
    3. Refuse invitations through HSAccess.refuse_invitation_to_group().
  3. Meanwhile, if too much time passes, the original inviter can:
    1. List unaccepted invitations through HSAccess.get_group_invitations_sent_by_user()
    2. Uninvite the invited user via HSAccess.uninvite_user_to_group().

List of methods and categories¶

  • Managing users
    • User creation and management:
      • HSAccess.assert_user(): create a user and update user metadata.
      • HSAccess.get_user_metadata(): get a user record.
      • HSAccess.assert_user_metadata(): push a user record with changes.
      • HSAccess.get_user_print_name(): get the print name of a user.
    • User status
      • HSAccess.user_exists(): True if user is valid.
      • HSAccess.user_is_active(): True if the user is active and enabled.
      • HSAccess.user_is_admin(): True if the user is an administrator.
    • Miscellaneous:
      • HSAccess.get_users(): a list of all active users.
  • Managing groups
    • Creation and management:
      • HSAccess.assert_group(): register a group.
      • HSAccess.get_group_metadata(): read a group registration record.
      • HSAccess.assert_group_metadata(): make changes in a group registration record.
      • HSAccess.retract_group(): remove a group (not recommended).
      • HSAccess.get_group_print_name(): get the print name for a group.
    • Views of group membership
      • HSAccess.get_groups(): a list of all valid groups.
      • HSAccess.get_groups_for_user(): a list of groups to which a user belongs.
    • Status of a group
      • HSAccess.group_exists(): whether group is valid
  • Managing resources
    • Creation and update:
      • HSAccess.assert_resource(): register a resource or update resource registration.
      • HSAccess.get_resource_metadata(): get a registration record.
      • HSAccess.assert_resource_metadata(): post changes to a registration record.
      • HSAccess.get_resource_print_name(): get the print name of a resource.
    • Status of a resource
      • HSAccess.resource_exists(): whether resource is valid.
      • HSAccess.resource_accessible(): whether a resource is accessible to a user.
      • HSAccess.resource_is_immutable(): whether a resource is immutable and thus archival.
  • Access control
    • For resources:
      • Access status:
        • HSAccess.resource_is_owned(): True if resource is owned by a user.
        • HSAccess.resource_is_readwrite(): True of resource is read/write to a user.
        • HSAccess.resource_is_readable(): True if resource is readable to a user.
        • HSAccess.resource_is_readable_without_sharing(): True if resource is readable without sharing for a user.
        • HSAccess.get_user_privilege_over_resource(): return numeric privilege 1-4 for a user.
      • User access to resources:
        • HSAccess.share_resource_with_user(): make a resource accessible to a user.
        • HSAccess.unshare_resource_with_user(): remove access to a resource for a user.
        • HSAccess.get_resources_held_by_user(): list the resources accessible to a user (by any means).
      • Group access to resources:
        • HSAccess.share_resource_with_group(): make a resource accessible to a group.
        • HSAccess.unshare_resource_with_group(): remove access to a resource for a group.
        • HSAccess.get_resources_held_by_group(): list the resources available to a group.
    • For groups:
      • Access status:
        • HSAccess.group_exists(): whether group is valid
        • HSAccess.group_is_owned(): whether group is owned by a specified user
        • HSAccess.group_is_readwrite(): whether group is read/write to a specified user.
        • HSAccess.group_is_readable(): whether group is readable to a user; minimum privilege.
        • HSAccess.get_user_privilege_over_group(): get numeric privilege 1-3 for a group.
      • User access to groups:
        • HSAccess.group_is_owned(): whether group is owned by a specified user
        • HSAccess.group_is_readwrite(): whether group is read/write to a specified user.
        • HSAccess.group_is_readable(): whether group is readable to a user; minimum privilege.
      • Membership reporting
        • HSAccess.user_in_group(): True if user is in a given group.
        • HSAccess.groups_of_user(): a list of groups in which the user is a member.
      • New members
        • HSAccess.invite_user_to_group(): invite a user.
        • HSAccess.get_group_invitations_sent_by_user(): get all invitations.
        • HSAccess.uninvite_user_to_group(): retract an invitation.
        • HSAccess.get_group_invitations_for_user(): get all invitations.
        • HSAccess.accept_invitation_to_group(): accept an invitation.
        • HSAccess.refuse_invitation_to_group(): refuse an invitation.
        • HSAccess.share_group_with_user(): DEPRECATED: group membership without invitation.
        • HSAccess.unshare_group_with_user(): remove all access to a group for a user.
  • Resource organization (not yet implemented)
    • Tagging of resources
      • HSAccess.assert_tag(): make a new tag.
      • HSAccess.retract_tag(): destroy a tag and delete all uses.
      • HSAccess.assert_resource_has_tag(): tag a resource.
      • HSAccess.retract_resource_has_tag(): untag a resource.
      • HSAccess.get_tags(): get a list of all active tags.
      • HSAccess.get_resources_by_tag(): get a structure of resources, filed by tag.
    • Folders for resources
      • HSAccess.assert_folder(): make a new folder.
      • HSAccess.retract_folder(): destroy a folder and remove all links in the folder.
      • HSAccess.get_folders(): get a list of all folders.
      • HSAccess.assert_resource_in_folder(): put a resource into a folder.
      • HSAccess.retract_resource_in_folder(): remove a resource from a folder.
      • HSAccess.get_resources_in_folders(): list resources by folder.
  • Statistics
    • HSAccess.get_number_of_group_owners()
    • HSAccess.get_number_of_groups_of_user()
    • HSAccess.get_number_of_groups_owned_by_user()
    • HSAccess.get_number_of_resource_owners()
    • HSAccess.get_number_of_resources_held_by_user()
    • HSAccess.get_number_of_resources_owned_by_user()
  • Current user
    • HSAccess.get_uuid()
    • HSAccess.get_login()
